# 历史封卷快照仓 - 修改说明

## 功能概述

问卷关闭或过期时自动生成只读快照，固化当时的题目结构、问卷状态和统计结果，避免后续编辑覆盖历史现场，适合活动结案、数据复盘和过程留痕。

### 修改文件

#### QuestionnaireService.java

**新增功能**：
- `getAllQuestionnaires()` - 遍历检查过期问卷，自动触发过期处理
- `getQuestionnaire(String id)` - 获取单个问卷时检查过期状态
- `processExpiredQuestionnaire(String questionnaireId)` - 过期处理核心方法：
  1. 校验问卷状态为 active
  2. 校验已过截止时间
  3. 将问卷状态更新为 expired
  4. 生成过期快照（reason=expired）
- `updateQuestionnaire()` - 状态变更为 closed 时自动生成关闭快照

**关键逻辑**：
- 过期快照以 `"expired"` 作为去重 key，同一份问卷只生成一份过期快照
- 关闭快照以 `"closed_closed"` 作为去重 key，同一份问卷只生成一份关闭快照
- 手动快照以 `"manual_时间戳"` 作为 key，每次创建都是唯一的

## 快照类型说明

| 类型 | reason | 去重规则 | 触发时机 |
|------|--------|----------|----------|
| 手动关闭封卷 | `closed_closed` | 每个问卷 1 份 | 手动将状态改为 closed 并保存 |
| 到期自动封卷 | `expired` | 每个问卷 1 份 | 检测到 active 问卷已过截止时间 |
| 手动快照 | `manual_时间戳` | 不限制，每次都新建 | 用户手动点击创建 |

## 使用方式

### 自动触发

1. **关闭封卷**：在编辑页将问卷状态改为"已关闭"，保存后自动生成关闭封卷快照
2. **到期封卷**：访问首页或编辑页时，系统自动检测已过期的 active 问卷，将状态更新为 expired 并生成过期快照

### 手动操作

1. 进入问卷编辑页
2. 切换到"历史快照"标签
3. 点击"+ 创建快照"按钮，可随时创建当前状态的快照
4. 点击任意快照卡片，查看快照详情（只读）

### 入口

- 首页 → 问卷卡片 → "快照"按钮
- 编辑页 → "历史快照"标签页
- 快照列表 → 点击卡片进入快照详情页

## 数据存储

快照采用 JSON 序列化存储，独立于原问卷数据，确保：
- 后续修改题目不会影响历史快照
- 后续修改统计数据不会影响历史快照
- 即使原问卷被删除，快照数据仍然保留

## 修复记录

### 修复 1：手动快照重复创建问题

**问题**：同一问卷同一状态下无法重复创建手动快照，连续两次创建第二次返回"快照已存在"。

**原因**：所有快照都使用 `reason + "_" + status` 作为去重 key，手动快照也被去重了。

**修复**：
- 手动快照使用 `manual_ + 时间戳` 作为唯一 key，每次创建都是新的
- 只有自动快照（closed、expired）才做去重限制
- 前端展示时判断 `startsWith('manual_')` 统一显示为"手动快照"

### 修复 2：过期问卷不自动生成快照

**问题**：过期问卷不会自动转为过期快照。

**原因**：缺少定时或触发机制来检测过期并生成快照。

**修复**：
- 在 `getAllQuestionnaires()` 和 `getQuestionnaire()` 中增加过期检测
- 检测到已过期的 active 问卷时，调用 `processExpiredQuestionnaire()` 处理
- 先将问卷状态更新为 expired，再生成过期快照
- 过期快照以 `"expired"` 为 key 去重，只生成一份

### 修复 3：过期快照状态停留在 active

**问题**：过期快照生成后，问卷状态和快照固化状态仍停留在 active。

**原因**：之前只生成快照但没有更新问卷状态，且快照记录的是更新前的状态。

**修复**：
- `processExpiredQuestionnaire()` 方法中先更新问卷状态为 expired 并保存
- 然后再调用 `snapshotService.createSnapshot()` 生成快照
- 确保快照的 `status` 和 `originalStatus` 都记录为 expired
